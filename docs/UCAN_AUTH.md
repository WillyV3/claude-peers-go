# UCAN Capability Auth

How the trust system works in Sontara Lattice.

## Overview

Every request to the broker requires a UCAN (User Controlled Authorization Network) capability token. Tokens are Ed25519-signed JWTs that encode exactly what the holder is allowed to do. A root key on the broker signs all tokens. Child tokens can only have a subset of their parent's capabilities.

## How it works

```
Root Key (broker)
  signs -> Root Token (all capabilities)
    delegates -> Machine Token (peer-session: register, message, list, events)
    delegates -> Service Token (fleet-write: list, events, memory read+write)
    delegates -> CLI Token (cli: list, send, events)
```

Each token contains:
- `iss`: Issuer's public key (base64url-encoded Ed25519)
- `aud`: Audience's public key (who this token was issued to)
- `sub`: Always `"claude-peers"` (prevents token confusion)
- `cap`: Array of capability objects (`{"resource": "peer/list"}`)
- `prf`: SHA-256 hash of the parent token (delegation proof)
- `exp`: Expiry timestamp (24h default for delegated, 1 year for root)
- `machine`: Machine name (for health/quarantine checks)

## Capabilities

| Resource | What it gates |
|----------|--------------|
| `peer/register` | Register as a peer |
| `peer/heartbeat` | Keep-alive signal |
| `peer/unregister` | Deregister |
| `peer/set-summary` | Update work summary |
| `peer/list` | Discover other peers |
| `msg/send` | Send messages |
| `msg/poll` | Receive messages |
| `msg/ack` | Acknowledge message delivery |
| `events/read` | Read fleet events |
| `memory/read` | Read fleet memory |
| `memory/write` | Write fleet memory |
| `nats/subscribe` | Subscribe to NATS (reserved for future use) |

## Roles

Pre-defined capability sets for common use cases:

| Role | Capabilities | Use case |
|------|-------------|----------|
| `peer-session` | peer/*, msg/*, events/read, memory/read | Claude Code MCP sessions |
| `fleet-read` | peer/list, events/read, memory/read | Gridwatch, monitoring |
| `fleet-write` | peer/list, events/read, memory/read+write | Dream, supervisor |
| `cli` | peer/list, msg/send, events/read | CLI commands |

## Key files

All stored in `~/.config/claude-peers/`:

| File | Purpose | Permissions |
|------|---------|------------|
| `identity.pem` | Ed25519 private key (PKCS8 PEM) | 0600 |
| `identity.pub` | Ed25519 public key (PKIX PEM) | 0644 |
| `root.pub` | Fleet root public key (from broker) | 0644 |
| `token.jwt` | Current UCAN capability token | 0600 |

## Attenuation

When issuing a child token, every requested capability must exist in the parent token. You cannot escalate -- a `fleet-read` token cannot delegate `memory/write` to a child.

```
Root Token: [peer/*, msg/*, events/read, memory/read, memory/write, nats/subscribe]
  -> peer-session: [peer/*, msg/*, events/read, memory/read]  (OK: subset)
  -> fleet-read: [peer/list, events/read, memory/read]         (OK: subset)
  -> REJECTED: [peer/list, admin/delete]                        (FAIL: admin/delete not in parent)
```

## Validation chain

When the broker receives a request:

1. Extract `Authorization: Bearer <jwt>` header
2. Parse JWT, verify Ed25519 signature using the `iss` claim's public key
3. Check expiry (30s leeway for clock skew)
4. Check `sub` == `"claude-peers"`
5. If root token (`prf` is empty): verify `iss` matches broker's root public key
6. If delegated token (`prf` is set): look up parent in cache, verify child caps are subset
7. Check machine health (if quarantined: 403)
8. Check required capability for the endpoint (if missing: 403)

## Error responses

All auth errors return JSON:

| Status | Code | Meaning |
|--------|------|---------|
| 401 | `NO_AUTH` | Missing Authorization header |
| 401 | `INVALID_TOKEN` | Malformed or unparseable JWT |
| 401 | `BAD_SIGNATURE` | Signature verification failed |
| 401 | `TOKEN_EXPIRED` | Token has expired |
| 401 | `BAD_PROOF` | Delegation chain is invalid |
| 403 | `MISSING_CAPABILITY` | Token lacks required capability |
| 403 | `QUARANTINED` | Machine is quarantined due to security events |

## Token lifecycle

- **Root token**: 1 year TTL, generated by `claude-peers init broker`
- **Delegated tokens**: 24 hour TTL, generated by `claude-peers issue-token`
- **Refresh**: Re-run `issue-token` and `save-token` (manual for now)
- **Revocation**: Tokens expire naturally. For immediate revocation, rotate the root key (`claude-peers init broker` and re-issue all tokens)

## Commands

```bash
# Generate root keypair + token (broker only)
claude-peers init broker

# Generate machine keypair (client)
claude-peers init client http://<broker-ip>:7899

# Issue a scoped token (on broker, outputs JWT to stdout)
claude-peers issue-token /path/to/machine.pub peer-session

# Save a token (on client)
claude-peers save-token <jwt-string>
```
